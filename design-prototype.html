API Design Guidelines

Engineering / RFCs · last edited 2m ago · 3 collaborators

This document outlines our conventions for designing internal and public-facing APIs. The goal is consistency across services, predictable behavior for consumers, and easy-to-read documentation.

Naming conventions

Use snake_case for all JSON field names. URL paths use kebab-case. Resource names are always plural: /api/documents, not /api/document.

Authentication

All endpoints require authentication via Bearer token in the Authorization header. Session cookies are accepted for browser requests. The sync server uses JWT verification with JWKS rotation.

Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
Content-Type: application/json

Error responses

All errors return a consistent JSON shape:

{
  "error": "resource not found",
  "status": 404
}
  • Error messages are always lowercase
  • Status codes follow standard HTTP semantics
  • Rate-limited requests return 429 with a retry-after header
